# Data Pipeline

A data engineer at a multinational corporation is designing a new data pipeline architecture. The company needs to consolidate data from multiple transactional databases, enable data analysis for business intelligence teams, and ensure the production databases' performance isn't impacted. They need to determine the most appropriate architectural pattern for data flow and processing.

**What is the recommended approach?**

- A) Direct all analytics queries to the transactional databases to ensure real-time data access
- B) Land data in a data lake, prepare it, then move selected data to a data warehouse for reporting
- C) Store all data directly in the data warehouse and perform transformations during query runtime
- D) Create separate data marts for each business unit without a centralized data warehouse

## Answer

**B) Land data in a data lake, prepare it, then move selected data to a data warehouse for reporting**

This approach separates analytics from production databases (protecting performance), centralizes data, and follows the recommended lake-to-warehouse pattern for BI workloads.

## Why the other options are incorrect

- **A) Direct all analytics queries to the transactional databases to ensure real-time data access** — Running analytics directly on production databases impacts their performance, which violates the requirement to protect production database performance.
- **C) Store all data directly in the data warehouse and perform transformations during query runtime** — This ELT-only approach skips the data lake stage, limiting flexibility. Performing all transformations at query time increases costs and query latency, and doesn't leverage the benefits of a centralized raw data store.
- **D) Create separate data marts for each business unit without a centralized data warehouse** — Without centralization, data silos form, making cross-business-unit analysis difficult. This also increases operational overhead with duplicated pipelines and inconsistent data.
