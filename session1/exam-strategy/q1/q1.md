# Kinesis Jail

An Amazon Kinesis application is trying to read data from a Kinesis data stream. However, the read data call is rejected. The following error message is displayed: ProvisionedThroughputExceededException.

**Which combination of steps will resolve the error? (Select TWO)**

- A) Configure enhanced fan-out on the stream
- B) Enable enhanced monitoring on the stream
- C) Increase the size of the GetRecords requests
- D) Increase the number of shards within the stream to provide enough capacity for the read data calls
- E) Make the application retry to read data from the stream

## Answer

**D** and **E**

- **D) Increase the number of shards within the stream to provide enough capacity for the read data calls** — Adding shards increases the overall read throughput capacity of the stream.
- **E) Make the application retry to read data from the stream** — Retrying with exponential backoff handles transient throughput exceeded errors.

## Why the other options are incorrect

- **A) Configure enhanced fan-out on the stream** — Enhanced fan-out provides dedicated throughput per consumer using HTTP/2, but doesn't directly resolve a ProvisionedThroughputExceededException caused by exceeding shard-level read limits when multiple consumers share the same shard.
- **B) Enable enhanced monitoring on the stream** — Monitoring provides visibility into stream metrics but does not resolve throughput issues. It helps diagnose problems, not fix them.
- **C) Increase the size of the GetRecords requests** — Larger GetRecords requests retrieve more records per call but don't increase the shard's throughput limit (2 MB/s per shard). This could actually worsen the problem by consuming more of the available throughput per request.
