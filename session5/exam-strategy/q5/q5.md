# S3 Dual-Layer Encryption

A healthcare company needs enhanced security for storing sensitive patient data in S3. Compliance requires multiple layers of encryption for all data at rest. The engineer wants to enforce dual-layer encryption for all objects uploaded to an S3 bucket.

Which combination of services and configurations would meet these requirements?

- A) Configure SSE-S3 encryption with S3 Bucket Keys enabled.
- B) Use client-side encryption with AWS KMS keys before uploading to S3.
- C) Implement DSSE-KMS encryption using AWS KMS keys in the same region as the bucket.
- D) Enable both SSE-KMS and SSE-S3 encryption on the bucket simultaneously.

## Answer

**C**

- **C) Implement DSSE-KMS encryption using AWS KMS keys in the same region as the bucket.**
  - DSSE-KMS is purpose-built dual-layer server-side encryption
  - Two independent layers of encryption applied automatically
  - KMS keys in same region required for DSSE-KMS
  - Can be enforced via bucket policy — meets compliance

## Why the other options are incorrect

- **A) Configure SSE-S3 encryption with S3 Bucket Keys enabled.**
  - SSE-S3 is single-layer encryption only
  - Bucket Keys optimize KMS costs but don't add layers
  - Doesn't meet dual-layer encryption requirement
  - Only one encryption layer applied
- **B) Use client-side encryption with AWS KMS keys before uploading to S3.**
  - Client-side + server-side could provide two layers
  - But this is client-side only — not enforced at bucket level
  - Cannot enforce via bucket policy alone
  - Operational overhead managing client-side encryption
- **D) Enable both SSE-KMS and SSE-S3 encryption on the bucket simultaneously.**
  - S3 doesn't support enabling two SSE types simultaneously
  - Must choose one server-side encryption method per object
  - This configuration is not technically possible
  - Invalid approach — S3 rejects conflicting encryption settings
